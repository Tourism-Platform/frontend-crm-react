# Проект: Tourism-Platform / Frontend-CRM

## Технологический стек
- React + TypeScript + Vite
- Styling: TailwindCSS
- Components: shadcn/ui

## Принципы разработки
- Соблюдение принципов SOLID, DRY и KISS.
- Использование функциональных компонентов и хуков.
- Чистка неиспользуемого кода.
- Акцент на чистоту и поддерживаемость.

## Правила взаимодействия
- Использовать русский язык.
- Краткость ответов (до 300 слов).
- Избегать избыточного кода.
- Использовать деструктуризацию пропсов в компонентах.
- Разбивать большие файлы на подкомпоненты.

## Специфичные правила
- **Архитектура FSD**: Следовать строгому разделению на слои и сегменты.
- **Структура виджетов/фич**:
  - `model/`: Содержит логику. Подпапки: `config/`, `handlers/`, `helpers/`, `types/`, `test/`.
  - `ui/`: Содержит визуальные компоненты. Большие компоненты выносить в подпапки внутри `ui/`.
- **Интернационализация (i18n)**:
  - Использовать JSON файлы в `public/locales/[lang]/`.
  - Каждая страница/крупный блок должен иметь свой отдельный JSON файл.
  - Все новые переводы должны быть типизированы в `src/shared/config/i18n/` (см. `i18n.types.ts` и `i18n.blocks.ts`).
- **Формы и валидация**:
  - Использовать кастомные поля из `shared/ui/custom/` (например, `CustomField`, `CustomRangeField`).
  - Всегда определять схемы валидации через **Zod**.
  - **Конфигурация полей**: Выносить список полей формы в `model/config/form.config.ts` с типизацией из `model/types/form.types.ts`. Это позволяет отрисовывать формы через маппинг в UI.
  - **Критично**: Все сообщения об ошибках в Zod должны использовать ключи i18n (например, `z.number({ message: "path.to.error" }).min(1, "path.to.error")`). Не оставлять дефолтные ошибки, чтобы избежать стандартных текстов zod или непереведенного текста.
- **Стандарты кода**: 
  - Следовать правилам именования и стиля, заданным в `eslint.config.js`.
  - Использовать чистые функции в `helpers`.
  - Не использовать any и never
  - Не использовать консоль для отладки, создания или удаления файлов
